### **阶段一：奠定基础 - 项目初始化与核心架构**

这个阶段的目标是搭建一个稳固的、可扩展的应用程序骨架。

* **步骤 1.1：初始化项目脚手架**
    * **做什么**：使用命令行，通过 `Electron React Boilerplate (ERB) + TypeScript` 模板创建一个全新的项目。
    * **目的**：为了获得一个预先配置好的、行业标准化的开发环境。ERB 已经处理了主进程与渲染器进程的构建、热重载、打包等复杂问题，让我们可以直接专注于业务逻辑开发，而不是耗费时间在环境配置上。

* **步骤 1.2：安装并配置数据持久化工具**
    * **做什么**：在项目目录下运行 `npm i electron-store` 来安装依赖。
    * **目的**：为应用程序提供本地数据存储能力。用户的会话历史、AI 配置、布局偏好等都需要在应用关闭后被记住，`electron-store` 提供了一个简单可靠的 JSON 文件存储方案，这是实现状态记忆的基础。

* **步骤 1.3：建立安全的进程间通信 (IPC) 桥梁**
    * **做什么**：
        1.  打开 `src/main/preload.ts` 文件。
        2.  定义一个 `Channels` 类型，列出所有将要用到的通信频道名称，如 `parallelchat/broadcast`、`parallelchat/store/get` 等。
        3.  使用 `contextBridge.exposeInMainWorld` 函数，将一个包含 `invoke`, `send`, `on` 等方法的 API 对象暴露给渲染器进程，并挂载到 `window.parallelchat` 上。
        4.  创建 `src/renderer/preload.d.ts` 类型声明文件，让 TypeScript 知道 `window.parallelchat` 的存在及其类型。
    * **目的**：**安全**。这是 Electron 开发的核心原则。通过预加载脚本和 `contextBridge`，我们可以避免在渲染器进程中开启 Node.js 集成，从而防止网页内容（渲染器）执行危险的本地代码。这种方式创建了一个定义严格、权限有限的“API”供渲染器调用，主进程可以对通过这个API的请求进行校验，极大地提升了应用的安全性。

* **步骤 1.4：在主进程中实现数据存储接口**
    * **做什么**：
        1.  在 `src/main/main.ts` 中，初始化 `electron-store`。
        2.  使用 `ipcMain.handle` 为 `parallelchat/store/get` 和 `parallelchat/store/set` 这两个频道编写处理器函数。这些函数将负责实际调用 `store.get()` 和 `store.set()` 方法。
    * **目的**：将所有文件系统的读写操作**集中**在主进程中管理。这是一种良好的架构实践，可以避免多个进程同时读写同一个文件导致的数据损坏或冲突。渲染器进程只负责发送“请求”，而主进程作为唯一的“数据管理员”来执行操作，确保了数据的一致性和安全性。

### **阶段二：构建骨架 - UI 界面与视图管理**

这个阶段的目标是创建用户能看到和交互的静态界面，并实现动态加载 AI 页面的核心能力。

* **步骤 2.1：搭建应用主框架 (App Shell) UI**
    * **做什么**：在 React 中，创建一个名为 `AppShell` 的组件。使用 CSS Flexbox 或 Grid 将其划分为左、中、下三个区域，分别作为侧边栏、主工作区和全局输入栏的容器。目前只需创建静态布局，不需要填充内容。
    * **目的**：构建应用的视觉基础。这个静态框架定义了各个功能组件的“位置”，为后续填充动态内容提供了舞台。

* **步骤 2.2：在主进程中实现 AI 视图 (WebContentsView) 创建函数**
    * **做什么**：在主进程中编写一个 `createAiView` 函数。该函数接收 AI 配置作为参数，返回一个创建好的 `WebContentsView` 实例。函数内部必须实现：
        1.  使用 `session.fromPartition('persist:<id>')` 为每个视图创建独立的会话，确保 Cookie 和登录状态隔离。
        2.  通过 `webRequest.onBeforeSendHeaders` 修改请求头，将 User-Agent 设置为普通浏览器，以防被网站识别为自动化工具。
        3.  监听 `did-finish-load` 事件，在页面加载完成后注入并执行 `delete navigator.webdriver;` 代码，移除自动化特征。
    * **目的**：这是应用能够集成第三方AI网站的核心技术。**隔离会话**是让用户能同时登录多个不同账号的关键。**伪装 User-Agent 和移除 webdriver 特征**则是为了绕过一些网站的反爬虫和反自动化机制，确保页面能正常加载和交互。

* **步骤 2.3：构建设置页面和 AI 管理功能**
    * **做什么**：在 React 中创建“设置页面”和“添加 AI 对话框”的组件。为这些组件中的按钮（如“保存”、“添加”）添加事件处理，使其调用 `window.parallelchat.invoke('parallelchat/store/set', 'aiProviders', ...)` 来更新存储在 `electron-store` 中的 AI 列表。
    * **目的**：为用户提供一个管理其AI助手列表的图形化界面。这使用户能够根据自己的需求定制应用，而不是硬编码固定的AI服务。

### **阶段三：注入灵魂 - 实现核心交互功能**

这个阶段的目标是让应用“活”起来，实现布局切换和核心的广播功能。

* **步骤 3.1：在主进程中实现工作区布局逻辑**
    * **做什么**：为主进程添加 `parallelchat/layout/set` 和 `parallelchat/workspace/bounds` 等 IPC 监听器。当收到渲染器发来的消息时（例如“切换到网格布局”），主进程负责计算每个 `WebContentsView` 的新坐标和尺寸 (`bounds`)，调用 `view.setBounds()` 应用变更，并通过 `contentView.addChildView/removeChildView` 管理叠加与切换。
    * **目的**：将视图布局的**控制权**保留在主进程。因为 `WebContentsView` 不属于渲染器的 DOM 树，渲染器无法直接控制其位置。这种“渲染器请求，主进程执行”的模式是 Electron 中管理子视图的标准做法。

* **步骤 3.2：创建 AI 网站的“遥控器” (AI Handlers)**
    * **做什么**：创建 `src/main/ai-handlers/` 目录。根据 `LLM网站处理.md` 文件中的代码片段，为每个 AI 创建一个 `.ts` 文件（如 `gemini.ts`），其中导出一个名为 `sendMessage` 的函数。然后在主进程中将这些模块导入，并存为一个对象，方便按 AI 的 ID 查找。
    * **目的**：将针对不同网站的 DOM 操作逻辑**模块化**和**解耦**。每个网站的输入框、发送按钮的选择器都不同，将它们分离开来，使得未来如果某个网站改版，我们只需要修改对应的 handler 文件，而不用动主逻辑，极大地提高了代码的可维护性。

* **步骤 3.3：实现全局输入栏和广播功能**
    * **做什么**：构建“全局输入栏”React 组件。当用户点击发送或按 `Ctrl+Enter` 时，组件会获取输入框的文本和选中的 AI 列表，然后调用 `window.parallelchat.invoke('parallelchat/broadcast', payload)`。主进程中的 `broadcast` 处理器会遍历目标 AI，从 `WebContentsView` 注册表中取出对应视图，通过中心化的 `getInjectionScript` 生成注入脚本，使用 `webContents.executeJavaScript()` 在目标视图中执行，并聚合成功/失败结果返回渲染器。
    * **目的**：实现应用的**核心价值**——“一次提问，多端响应”。这个流程打通了从用户输入到最终在多个AI网站上自动执行发送操作的完整链路。

### **阶段四：提升体验 - 会话管理与新用户引导**

这个阶段的目标是增加应用的实用性和友好度，让其从一个“工具”变为一个“产品”。

* **步骤 4.1：实现会话侧边栏的状态管理**
    * **做什么**：构建“会话侧边栏”React 组件。
        * **保存会话**：当在一个“新会话”状态下首次发送消息后，通过 IPC 获取所有 `WebContentsView` 的当前 URL，并将它们作为一个新的会话对象保存到 `electron-store` 中。
        * **切换会话**：点击某个会话时，从 `electron-store` 读取该会话保存的 URL 列表，通过 IPC 发给主进程，主进程再命令对应的 `WebContentsView` 加载这些 URL。
    * **目的**：实现**上下文的保存与恢复**。这让用户可以中断一个复杂的对话，切换到另一个任务，然后再无缝地回来继续，极大地提升了应用的生产力价值。

* **步骤 4.2：创建新用户引导 (Onboarding) 流程**
    * **做什么**：在主进程启动时，检查 `electron-store` 中是否存在 `hasOnboarded` 标志。如果不存在，就向渲染器发送一个 `parallelchat/onboarding/start` 消息。渲染器收到后，依次弹出欢迎窗口、强制引导用户添加第一个 AI、显示一个指向全局输入栏的一次性高亮提示（Coach Mark），最后通知主进程设置 `hasOnboarded` 标志。
    * **目的**：**降低新用户的学习成本**。一个好的引导流程可以帮助用户在几分钟内理解应用的核心价值并完成基本设置，从而避免用户因困惑而流失，这对于提升用户初次体验和留存率至关重要。

* **步骤 4.3：最终打磨与测试**
    * **做什么**：实现一些细节交互，如侧边栏的拖拽调宽功能。严格按照 TDD 文档中的“验证步骤建议”，系统性地测试所有功能组合，确保没有明显的 Bug。
    * **目的**：**保证产品质量**。通过系统性的测试来发现并修复潜在问题，确保交付给用户的产品是稳定、可靠且符合预期的。
